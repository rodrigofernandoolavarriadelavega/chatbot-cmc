## Implementación técnica

### En el bot (flow de respuesta):

1. **Hook nuevo en `app/admin_routes.py`** — endpoint `POST /admin/winback/lanzar`:
   - Body: `{cohorte: "MG"|"PODO"|"KINE"|"NUTRI"|"MUJER", limit: 50, dry_run: true}`
   - Lee lista de teléfonos de `medilink_pagos_full.csv` o tabla del heatmap
   - Filtra: ultima_cita < 2025-10-01 AND profesional in [...] AND opt_in=true
   - Encola envíos en cola con rate limit 30msg/min para no caer en spam
   - Loguea en tabla `winback_envios`

2. **Detección de respuestas** en `flows.py`:
   - Si paciente responde "AGENDAR" / "PODO" / "KINE" / "NUTRI" / "MUJER" → entra a flow de agendamiento de la especialidad correspondiente
   - Si responde "BAJA" → setear `opt_in=false` y confirmar dado de baja
   - Métrica: log `winback_respondido` / `winback_agendo` / `winback_baja`

3. **Tabla `winback_envios`:**
```sql
CREATE TABLE winback_envios (
  id INTEGER PRIMARY KEY,
  phone TEXT,
  cohorte TEXT,
  template_id TEXT,
  enviado_ts INTEGER,
  respondio INTEGER DEFAULT 0,
  respondio_ts INTEGER,
  agendo INTEGER DEFAULT 0,
  agendo_ts INTEGER,
  baja INTEGER DEFAULT 0
);
```

4. **Dashboard de win-back:** agregar tab al dashboard `/camino-50m` con cohorte × enviados × respondidos × agendados × tasa.

## Riesgos y mitigación

1. **Templates rechazados por Meta** — si copy es muy "marketing-ish". Mitigación: variantes UTILITY más neutras pre-aprobadas.
2. **Quejas / bajas masivas** — si el mensaje se siente spammy. Mitigación: nunca más de 1 mensaje cada 90 días al mismo phone, opt-in claro, baja inmediata.
3. **Bot saturado** si 50+ responden simultáneo. Mitigación: rate limit del envío (30/min) + monitoreo cola.
4. **Paciente cambió de número** — bounces 3-5%. Mitigación: log de fallos para limpiar base.
5. **Cumplimiento Ley 21.719** — opt-out claro, registro de bajas, no enviar si está en blacklist. Mitigación: `opt_in` y `opt_out_ts` en tabla pacientes; respetar.
